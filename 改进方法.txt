# QQ智能助手架构重构方案
## 一、现状分析与差距识别
基于架构评审报告，当前系统在5个维度存在结构性缺陷：

维度 现状 目标 差距 高并发 SQLite单写者 + threading.Lock混用 + 无缓存，实际QPS <5 100+ QPS 20倍差距 高可靠 全栈单点故障 + 无故障转移 + 消息丢失 99.9%可用性 无冗余、无自愈 高安全 5层防线4层失效 + HTTP明文 + SQL注入风险 纵深防御+零信任 安全防线需重建 工程实践 57个手动代理 + 全局单例 + 零测试 + 无CI/CD 可测试/可维护/可扩展 需系统性重构 数据架构 SQLite单文件 + Faiss内存索引 + 三套连接模式 读写分离 + 持久化向量库 + 统一访问层 需迁移

硬件约束 ：单RTX 3090 24GB + 14 vCPU + 90GB RAM + 30GB系统盘。vLLM + Qwen2.5-7B AWQ量化在10并发下约3-5 QPS，这是GPU推理的物理天花板。100 QPS目标需通过 缓存+异步+队列+降级 组合策略实现。

## 二、目标架构概览
架构风格 ： 模块化单体（Modular Monolith） ——单FastAPI进程内按领域划分模块，模块间通过接口（Protocol/ABC）通信，未来可拆分为微服务。

## 三、核心设计决策
### 3.1 架构风格：模块化单体
选项 优势 劣势 结论 微服务 独立部署/扩展 运维复杂度高，单GPU无扩展收益 不适合 单体 简单 耦合严重，当前状态 需改进 模块化单体 模块边界清晰+部署简单 需纪律性维护模块边界 选择

选择理由 ：单GPU服务器无法水平扩展推理层，微服务的独立扩展优势无法体现。模块化单体在保持单进程部署简单性的同时，通过领域模块+接口隔离实现高内聚低耦合，未来需要时可沿模块边界拆分。

### 3.2 技术栈推荐
层次 当前 目标 迁移理由 数据库 SQLite PostgreSQL 16 + pgvector MVCC并发、JSONB、统一向量+关系查询 向量库 Faiss (内存) Qdrant (自托管) 持久化、payload过滤、HNSW高性能、API友好 缓存/队列 无 Redis 7 (Streams+语义缓存) 消息队列+缓存+限流+会话，一石四鸟 推理引擎 vLLM (单实例) vLLM + AWQ量化 + Multi-LoRA 10并发3-5QPS，8-12个活跃LoRA 前端代理 57个手动route.ts Next.js rewrites + Server Actions 消除重复代码，Cookie安全代理 认证 JWT localStorage JWT httpOnly Cookie + RS256 防XSS窃取，非对称签名 数据库迁移 _init_tables() Alembic 版本化迁移，可回滚 容器化 无Dockerfile Docker Compose + Dockerfile 可重复部署

### 3.3 数据架构
读写分离策略 ：

- 写路径 ：消息记录 → PostgreSQL（WAL归档）+ Redis Streams（异步写入，最终一致性）
- 读路径 ：热点查询 → Redis缓存 → PostgreSQL回源
- 向量检索 ：Qdrant（独立进程，gRPC通信），元数据从PostgreSQL同步
数据一致性 ：

- 消息写入：PostgreSQL为主，Redis Streams异步分发，消费者确认后标记完成
- 向量索引：文档写入PostgreSQL后，通过事件触发Qdrant索引更新（最终一致性，<1秒延迟）
### 3.4 服务通信方式
场景 方式 理由 前端↔后端 HTTP REST (Next.js rewrites代理) 简单，BFF模式 后端↔vLLM HTTP OpenAI兼容API vLLM原生支持，LoRA路由通过model字段 后端↔Qdrant HTTP/gRPC Qdrant原生支持 后端↔Redis redis.asyncio 原生异步，P95延迟18ms 模块间通信 Python Protocol/ABC接口 模块解耦，可mock测试 异步任务 Redis Streams (Consumer Group) 消息持久化、消费者组负载均衡

## 四、并发与性能优化策略
### 4.1 目标QPS拆解
核心思路 ：100 QPS不意味着100次LLM推理。通过多级过滤，将实际LLM调用量降至3-5 QPS。

### 4.2 多级缓存架构
缓存层 命中率 延迟 容量 L1 进程内LRU 10-15% <1ms 1000条 L2 Redis语义缓存 40-60% ~5ms 10万条 L3 vLLM前缀缓存 系统提示100%复用 TTFT降50% GPU VRAM L4 完整推理 剩余 1-5s -

### 4.3 异步处理管道
关键设计 ：

- 消息接收与推理 完全解耦 ：NoneBot2只负责收发消息，推理请求入Redis Streams
- 优先级调度：紧急消息（含"急"/"紧急"）优先级0，闲聊优先级10
- 超时降级：推理超过8秒返回预设回复"我正在思考中，请稍等..."
- 背压机制：队列长度超过500时，新消息直接返回降级回复
### 4.4 vLLM优化配置
预期性能 ：

- 单请求：78 tok/s，128 tokens约1.6秒
- 10并发：38 tok/s，128 tokens约3.4秒
- 理论推理QPS：3-5（10并发下）

总结 ：本方案以模块化单体为架构风格，通过Redis(缓存+队列+限流)、PostgreSQL+pgvector(统一数据+向量)、Qdrant(高性能检索)、vLLM AWQ+Multi-LoRA(高效推理)四大中间件，在单RTX 3090服务器上实现100 QPS目标。核心策略是 多级缓存过滤+异步队列解耦+降级链保底 ，将实际LLM调用量从100 QPS降至3-5 QPS。安全层面重建5层纵深防御，可靠性层面实现降级链+自动恢复+数据备份。迁移分4阶段渐进式推进，每阶段可独立回滚。
